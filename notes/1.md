The repo converts human egocentric demos into robot-trainable data by turning the human hand into a robot-gripper-like end-effector trajectory: 6D pose plus open/closed grasp, aligned with object poses and optional clean visual input.

**Pipeline**
1. **Start from Project Aria data**
   Raw input is `sample.vrs` plus MPS `slam/` and `hand_tracking/`. The README says preprocessing “turns raw MPS output into training-ready data” via `python -m preprocess.Preprocess` ([README.md](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/README.md:231)).

2. **Extract human hand as robot-like kinematics**
   `AriaHands.py` processes MPS hand tracking into wrist/palm/midpoint poses, grasp states, and smoothed world-space trajectories. The key bridge is the “custom gripper-like midpoint frame between the thumb and index fingertips” ([AriaHands.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/AriaHands.py:15)). Grasp is detected from thumb-index fingertip distance ([AriaHands.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/AriaHands.py:214)).

3. **Estimate object poses**
   The preprocessing pipeline segments objects/arms, tracks object keypoints, and triangulates them into 3D object poses. The repo summary lists `dinosam`, `kptsselector`, `cotracker`, and `camtriangulator` before `datasetgen` ([preprocess/README.md](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/README.md:87)). `CamTriangulator` uses camera poses and 2D tracks to triangulate 3D points and object poses ([CamTriangulator.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/CamTriangulator.py:256)).

4. **Build `training_data.json`**
   `DatasetGen` is the final preprocessing stage. Its docstring says it consolidates RGB, masks, SLAM, hands, and object poses into `training_data.json` for Multi-Object Flow Matching policies ([DatasetGen.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/DatasetGen.py:9)). It writes:
   - `hands`: `T_hand_to_world` and `grasp`
   - `objects`: `T_obj_to_world` and dynamic flags
   - camera/image/mask metadata

   The exact write happens at [DatasetGen.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/DatasetGen.py:566).

5. **Handle manipulation under occlusion**
   When the human grasps an object, `DatasetGen` “latches” the closest non-anchor object to the hand and propagates the object pose by forward kinematics ([DatasetGen.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/preprocess/DatasetGen.py:461)). This makes object state stable even when the hand occludes the object.

6. **Convert per-frame JSON into model inputs and robot-style targets**
   `FlowMatchingDataloader` indexes every `all_data/<idx>/training_data.json` ([FlowMatchingDataloader.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/training/FlowMatchingDataloader.py:231)). It builds **ICT tokens** for hands and objects: hand tokens contain pose plus grasp, object tokens contain pose plus hand-relative geometry ([FlowMatchingDataloader.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/training/FlowMatchingDataloader.py:311)).

   Then it creates future action targets `y_action`: position, 6D rotation, and grasp for the future human hand trajectory ([FlowMatchingDataloader.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/training/FlowMatchingDataloader.py:446)). This is the robot-relevant target: not human joint angles, but Cartesian end-effector pose + gripper state.

7. **Train policy on those targets**
   The trainer loads `x_rgb`, `x_ict`, and `y_action`; if no object-dynamics auxiliary target is used, `x_1 = y_action` ([FlowMatchingTrainer.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/training/FlowMatchingTrainer.py:421)). So the model learns to predict future end-effector-like hand trajectories from image plus interaction tokens.

8. **Deploy by replacing “human hand” with robot EE**
   At inference, the robot end-effector pose is read as the same “hand” representation: `hands_in_cam = arms[s].get_T_ee_in_cam() @ T_align` ([run_inference.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/inference/run_inference.py:243)). The policy builds the same ICT format, predicts future hand trajectories, then decodes them into robot EE targets ([policy.py](/Users/lewis/SchoolWorks/06_Junior_Spring/research1/HumanEgo/inference/policy.py:316)).

So the short version: **human hand motion is retargeted implicitly by abstraction**. The repo converts the hand into a gripper-like SE(3) pose and grasp signal, trains a policy to predict that trajectory around objects, and at robot time treats the robot end-effector as the same “hand” token. No robot joint labels are needed in the human dataset; IK/control happens later during deployment.
