# HumanEgo — Preprocessing

Preprocessing turns a raw **Project Aria** recording (`sample.vrs` + MPS SLAM & hand
tracking) into a structured, per-frame dataset with 3D hand/object tracking, ready to
train HumanEgo policies. This doc covers (1) how to download a sample and run the
pipeline, (2) the pipeline architecture, and (3) every file it produces and what it means.

The data lives on the public HuggingFace dataset
**[`Leo-TX/HumanEgo`](https://huggingface.co/datasets/Leo-TX/HumanEgo)** — no token required.

---

## 1. Quick start

```bash
pip install huggingface_hub        # for downloading
# and follow the repo README to install the full preprocessing environment
```

**Download a sample recording** (`scripts/download_data.py`):

```bash
# (a) INPUT ONLY (~0.6 GB) — run the pipeline yourself and reproduce every output
python scripts/download_data.py --mode test

# (b) INPUT + PRECOMPUTED OUTPUT (~2 GB) — skip the GPU pipeline; all_data.tar auto-extracted
python scripts/download_data.py --mode full
```

Use `--out <dir>` to download somewhere other than `./data`.

**Run the pipeline:**

```bash
python -m preprocess.Preprocess \
    --mps_path ./data/serve_bread/aria/mps_serve_bread_000_vrs --task serve_bread
```

Everything is written to `…/mps_serve_bread_000_vrs/preprocess/`. On an RTX 4090 a
~33 s / 1000-frame recording takes ~20 min end-to-end.

<details>
<summary>Plain <code>huggingface_hub</code> (no helper script)</summary>

```python
from huggingface_hub import snapshot_download
# input only
snapshot_download(
    repo_id="Leo-TX/HumanEgo", repo_type="dataset", local_dir="./data",
    allow_patterns=["serve_bread/aria/mps_serve_bread_000_vrs/*"],
    ignore_patterns=["serve_bread/aria/mps_serve_bread_000_vrs/preprocess/*"],
)
```
Drop `ignore_patterns` to also fetch the precomputed output, then unpack the per-frame
archive: `tar -xf …/preprocess/all_data.tar -C …/preprocess`.
</details>

## 2. Data layout

```
mps_serve_bread_000_vrs/
├── sample.vrs              # raw Aria recording                     ┐
├── slam/                   # MPS SLAM (trajectory + semidense points)│ INPUT
├── hand_tracking/          # MPS hand tracking (hand_tracking_results.csv)
├── else/                   # VRS metadata / health checks           ┘
└── preprocess/             # ← everything below is GENERATED by the pipeline
    ├── <top-level files>   # per-stage results + diagnostic plots
    ├── vis/                # visualization clips (mp4 + gif)
    ├── cfg/                # snapshot of the configs used (auto-regenerated)
    └── all_data/<idx>/     # per-frame data (rgb, masks, poses, training_data.json)
```

## 3. Pipeline architecture

`preprocess.Preprocess` runs these stages in order (entry point: `preprocess/Preprocess.py`):

| # | Stage | What it does | Models |
|---|-------|--------------|--------|
| 1 | **aria** | Extract RGB frames + camera config (`AriaCam`), MPS hand keypoints (`AriaHands`), SLAM device poses (`AriaSlam`), and segment the sequence into **Navigation / Transition / Manipulation** phases (`AriaPhases`). | projectaria_tools (MPS) |
| 2 | **indices** | Isolate the **object-centric window** (the manipulation segment) used by all downstream 3D stages. | — |
| 3 | **dinosam** | Detect + segment the prompted objects (`obj1`, `obj2`) and the arm/hands, per frame → object & arm masks. | Grounding DINO + SAM 2 |
| 4 | **kptsselector** | Pick trackable keypoints on each object. | — |
| 5 | **cotracker** | Track the 2D keypoints across the object-centric frames. | CoTracker3 |
| 6 | **camtriangulator** | Lift the 2D tracks to **3D object poses** using the SLAM camera poses. | — (geometry) |
| 7 | **lama** | Inpaint the arm/hand out of the RGB frames (arm-free frames). | LaMa |
| 8 | **visualkpts** | Render arm + object keypoints onto the RGB and arm-inpainted RGB. | — |
| 9 | **datasetgen** | Consolidate everything into per-frame `training_data.json` — 4×4 SE(3) poses for cameras / hands / objects in an **object-centric world frame**, using a grasp "latch & propagate" state machine to keep object pose stable under occlusion. Also renders the object-centric 3D scene. | — |

> The released default config runs hand tracking via **aria_mps only**. The
> mediapipe / wilor / hamer methods are an **ablation** and are off by default (enable them
> in `cfg/preprocess/base/Preprocess.yaml` → `hand_tracking_methods` if you want to
> reproduce the hand-tracking ablation; they require extra model checkpoints).

## 4. Output reference — every file

### 4.1 Top-level `preprocess/` files

| File | Stage | Meaning |
|------|-------|---------|
| `aria_cam_rgb_config.json` | aria | RGB camera calibration/config (intrinsics, resolution, fps). |
| `aria_phases_results.json` | aria | Phase segmentation: frame ranges for Navigation/Transition/Manipulation + the object-centric window. |
| `aria_phases_analysis.png` | aria | Plot of the phase segmentation over time. |
| `aria_slam_analysis.png` | aria | Plot of the SLAM trajectory / pose analysis. |
| `aria_hands_analysis_l.png` / `_r.png` | aria | Left / right hand tracking analysis plots. |
| `dinosam_mask_obj1.png` / `obj2.png` | dinosam | DINO+SAM2 segmentation preview for object 1 / object 2. |
| `kptsselector_results.json` | kptsselector | The keypoints selected on each object for tracking. |
| `kptsselector_vis_obj1.png` / `obj2.png` | kptsselector | Visualization of the selected keypoints. |
| `cotracker_results.json` | cotracker | 2D CoTracker tracks of the keypoints across the object-centric frames. |
| `camtriangulator_results.json` | camtriangulator | Triangulated 3D object keypoints/poses per frame. |
| `camtriangulator_vis.ply` | camtriangulator | 3D point cloud of the triangulated keypoints. |
| `camtriangulator_vis.png` | camtriangulator | 2D diagnostic render of the triangulation. |
| `object_centric.ply` | datasetgen | Final object-centric 3D scene (multi-object point cloud + trajectories). |
| `object_centric.png` | datasetgen | 2D diagnostic plot of the object-centric scene. |

### 4.2 `preprocess/vis/` — visualization clips (`.mp4` + `.gif`)

| File | Meaning |
|------|---------|
| `aria_cam_rgb.*` | The raw extracted RGB stream. |
| `aria_vis.*` | RGB + MPS hand keypoints + phase labels over the sequence. |
| `aria_vis_full_hands.*` | RGB with full hand skeletons. |
| `visualkpts_vis.*` | Tracked arm + object keypoints on (arm-inpainted) RGB. |

### 4.3 `preprocess/all_data/<idx>/` — per-frame data

`<idx>` is the zero-padded frame index. Image-name convention: **W**=with, **Wo**=without,
**Arm**/**Obj**=arm/object, **Kpts**=keypoints overlaid.

| File | Meaning |
|------|---------|
| `rgb.png` | Extracted RGB frame. |
| `rgb_WoArm.png` | RGB with the arm inpainted out (LaMa). |
| `rgb_WArmObjKpts.png` | RGB with arm + object keypoints overlaid. |
| `rgb_WoArm_WArmObjKpts.png` | Arm-inpainted RGB with arm + object keypoints overlaid. |
| `mask_obj1.png` / `mask_obj2.png` | Per-frame object segmentation masks. |
| `mask_arm.png` | Per-frame arm/hand segmentation mask. |
| `mask_arm_and_obj.png` | Combined arm + object mask. |
| `aria_cam_rgb.json` | Per-frame camera pose (`c2w`) + intrinsics. |
| `aria_hands.json` | Per-frame MPS hand keypoints / pose. |
| `aria_slam.json` | Per-frame SLAM device pose. |
| `aria_phases.json` | Per-frame phase label. |
| `training_data.json` | **The per-frame training target** (schema below). Written for object-centric frames. |

### 4.4 `training_data.json` schema

The final per-frame dataloader target produced by `DatasetGen`:

```jsonc
{
  "metadata": {
    "idx", "ts",                 // frame index, timestamp
    "w", "h", "fps",             // image size, frame rate
    "k":   [3×3 intrinsics],
    "c2w": [4×4 camera-to-world],
    "anchor_key": "obj1"         // object designated as the world origin (object-centric frame)
  },
  "obs": {                       // relative paths to this frame's images
    "rgb_path", "mask_arm_path", "mask_obj_path",
    "rgb_WoArm_path", "rgb_WArmObjKpts_path", "rgb_WoArm_WArmObjKpts_path", ...
  },
  "entities": {
    "hands":   { "right": { "T_hand_to_world": [4×4 SE(3)], "grasp": <float> } },
    "objects": { "obj1": { "T_obj_to_world": [4×4], "is_dynamic": <bool> },
                 "obj2": { ... } },
    // ablation only (empty unless mediapipe/wilor/hamer were enabled):
    "hands_mediapipe": {...}, "hands_wilor": {...}, "hands_hamer": {...}
  }
}
```

### 4.5 `preprocess/cfg/`

A snapshot of the merged YAML configs used for this run, one file per module
(`AriaCam.yaml`, `DINOSAM.yaml`, …). **Auto-generated every run and fully regenerable** —
it documents exactly which settings produced the outputs.

---

## 5. Configuration

Configs live under `cfg/preprocess/`:

- `cfg/preprocess/base/Preprocess.yaml` — the master config; recursively loads each module's
  YAML and sets `hand_tracking_methods`.
- `cfg/preprocess/base/<Module>.yaml` — per-module settings (AriaCam, DINOSAM, CoTracker, …).
- `cfg/preprocess/tasks/<task>.yaml` — per-task overrides selected by `--task`, e.g.
  `serve_bread.yaml` sets the DINO prompts (`obj1: "piece of bread"`, `obj2: "a plate"`) and
  which hand to track.

```bash
python -m preprocess.Preprocess --mps_path <recording_dir> --task <task> \
    [--cfg_path cfg/preprocess/base/Preprocess.yaml] [--no-video] [--no-gif]
```

## 6. Downloading more data

```bash
# everything, but skip the large per-frame all_data.tar archives
python scripts/download_data.py --mode all

# everything, including all_data.tar (very large), auto-extracted
python scripts/download_data.py --mode all --with-tar
```

`all_data.tar` is just the per-frame files packed into one archive (so the dataset stays at
a few dozen files per recording instead of hundreds of thousands). It is fully regenerable by
running preprocessing, so you only need it to get the precomputed per-frame results without
running the pipeline.
